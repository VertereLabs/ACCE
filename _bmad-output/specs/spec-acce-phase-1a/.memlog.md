- (decision) Why: group revision classes are ACCE's highest-value, simplest product (one tutor, 4-6 paid seats per fixed 2hr slot, R290/seat). Live Test-3 demand now. Turns the manual WhatsApp funnel into online paid seats. Must never oversell a scarce class.
- (capability) CAP-1 Passwordless auth: a student signs up/logs in via magic link (no password). success: submitting email sends a magic link; clicking a valid link creates a session and lands in the portal; expired/used link is refused.
- (capability) CAP-2 Role-gated areas: STUDENT and ADMIN roles; portal requires student, admin requires ADMIN, marketing stays public. success: an unauthenticated visitor to a portal route is redirected to sign-in; a non-admin hitting an admin route/action is refused with no admin data leaked; guards enforced server-side at every data/mutation entry (spine AD-3), not just layout.
- (capability) CAP-3 Browse classes with live seats-left: a student sees upcoming SCHEDULED classes each showing subject, date/time, price, and N seats left (capacity minus PENDING+CONFIRMED, expired holds excluded). success: a class at/over capacity shows 'Class full' instead of an active Book action; empty list shows an empty state; seats-left reflects reality without a stored counter (spine AD-5).
- (capability) CAP-4 Seat checkout (balance or Paystack): a student reserves a seat, paying from wallet balance (instant CONFIRMED) or via Paystack one-off (PENDING hold then webhook confirm). success: balance>=price creates CONFIRMED + one BOOKING_CHARGE with no Paystack; else a PENDING hold (15min) + Paystack init URL; all seat acquisition goes through one reserveSeat() (spine AD-4).
- (capability) CAP-5 No-oversell under concurrency: a full class cannot oversell even when several students buy the last seat at once. success: N+1 concurrent reservations at an N-seat class -> exactly N succeed, rest get 'class full'; guaranteed by Serializable SSI + bounded retry, proven by a real-Postgres concurrency test (spine AD-4, NFR1).
- (capability) CAP-6 Reliable, idempotent payment confirmation: a paid seat is confirmed exactly once by a verified webhook. success: the webhook rejects a bad HMAC-SHA512 signature; a repeated/concurrent reference confirms once (Payment.reference is the idempotency gate) and writes exactly one BOOKING_CHARGE; the browser callback never confirms (spine AD-7, AD-8).
- (capability) CAP-7 Wallet ledger + view: a student sees balance and read-only history; balance is the running sum of an append-only signed ledger. success: wallet page shows current balance + chronological entries; R0 shows an empty state; every mutation appends an immutable row via the single per-student-locked wallet.mutate and can never drive balance negative (spine AD-6).
- (capability) CAP-8 Join-details gating: the Meet link (online) or location (in-person) is revealed only to CONFIRMED students. success: a non-confirmed viewer never receives meetingUrl/location in the payload (omitted server-side, not just hidden); a confirmed student sees it on the detail page and in the confirmation email (spine AD-10).
- (capability) CAP-9 Cancellation with tiered refund to wallet: a student cancels and is refunded to wallet by hours-to-start tier, freeing the seat. success: refund% computed server-side (h>=48 100%, 24<=h<48 70% with 30% fee, h<24/no-show 0%), CANCELLATION_REFUND (+CANCELLATION_FEE) written, enrollment CANCELLED, seat returns to pool; refund/fee is one decomposition (fee=price-refund) (spine AD-11).
- (capability) CAP-10 Admin class management: an admin creates/edits/lists group classes (subject, level, date/time, capacity, per-seat price, mode/location, manual Meet link). success: valid input persists a SCHEDULED GroupSession (price in cents); invalid input is rejected with field errors; capacity cannot drop below occupied seats (checked under the seat lock); concurrent admin edits are guarded by optimistic updatedAt (spine AD-16).
- (capability) CAP-11 Roster & attendance: an admin sees per-class enrollment (paid vs pending) and marks ATTENDED/NO_SHOW. success: roster lists each enrolled student with status in a token-styled table; empty class shows an empty state; attendance transitions go through enrollment.ts (sole status writer, spine AD-14).
- (capability) CAP-12 Seat-confirmation email: on a seat becoming CONFIRMED, the student is emailed the joining details. success: a Resend email (native fetch) with class, date/time, Meet link/location, and price is sent on confirmation via balance or webhook; an email-send failure does NOT roll back the confirmed seat (logged) (spine AD-13).
- (capability) CAP-13 Seeded real data: the DB is seeded with 4 subjects, levels, Priyanka as ADMIN, and the 6 Test-3 classes. success: after seed the portal shows the 6 real classes (2 Accounting, 2 Tax, 1 Auditing, 1 Management Accounting; 2hr, R290, cap 4-6); re-running the seed does not duplicate (idempotent).
- (capability) CAP-14 Late/orphan payment conserves money: a verified charge that arrives after the seat is lost credits the student's wallet rather than overselling or keeping the money. success: charge.success after hold-expiry+full -> captured amount credited to wallet balance (idempotent by reference), no oversell, no silent loss (spine AD-15; supersedes epics Story 4.2).
- (constraint) No-oversell is absolute: seat reservation runs in one Serializable transaction (SSI is the guarantee; GroupSession FOR UPDATE serialises writers) with bounded retry on P2034/40001; all acquisition through one reserveSeat(); enrollment.ts is the sole writer of every status transition (spine AD-4, AD-14).
- (constraint) Money integrity: all amounts integer cents (no floats); wallet is an append-only immutable ledger, balance = running sum, never negative, every mutation under a per-student lock via one wallet.mutate; exactly one BOOKING_CHARGE per confirmed enrollment (partial unique index) (spine AD-6, AD-8, AD-9).
- (constraint) Payment trust boundary: Paystack webhook verifies HMAC-SHA512 of the RAW body, is idempotent by Payment.reference (insert-as-gate), runs runtime=nodejs, and is the sole confirmer; the browser callback never confirms; external I/O via native fetch, no axios/SDK (spine AD-7, AD-13, NFR9).
- (constraint) Additive isolation: portal/admin are route groups in the SAME Next app (output:standalone, Coolify, base /acce-nextjs); marketing routes, SEO and headers untouched (headers only extended additively); DB access only via the src/lib/db.ts Prisma singleton (spine AD-1, AD-2, NFR6, NFR7, NFR8).
- (constraint) CPA-compliant wallet: unused balance must remain redeemable (ledger designed so it can be paid back out even before a withdraw button ships); late-cancel fee modelled as a distinct CANCELLATION_FEE service-fee event, not kept unspent money (STRATEGY legal note; spine AD-6).
- (constraint) Accessibility floor on every new portal/admin control: body text >=4.5:1 / large >=3:1 in both modes, keyboard-operable with visible focus ring, >=44px targets, honor prefers-reduced-motion; reuse existing navy+gold tokens + shadcn, no new palette (epics NFR10, DESIGN.md).
- (constraint) Operational floor: rate-limit magic-link send + webhook; structured logs + alerts on money paths; Paystack test/live keys env-split; DB backups for auditability (spine ops conventions).
