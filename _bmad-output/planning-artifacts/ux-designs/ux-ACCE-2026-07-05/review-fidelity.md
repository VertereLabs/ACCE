# Brand & Implementation Fidelity Review — ACCE

## Overall verdict

The frozen DESIGN.md is an unusually faithful spec: every mode-token triple in the frontmatter matches the approved mock `:root` blocks **exactly** (I diffed all ~18 tokens per mode against `mock-final-dark-default.html` / `mock-final-light-mode.html` and found zero numeric drift), the two-mode system is internally coherent, and the HSL-channel token format is identical to the app's existing shadcn convention — so the palette itself is a genuine values-swap. The problem is **not** the spec's tokens; it's that the *current* `acce-nextjs/globals.css` is not a clean shadcn baseline. It carries a hardcoded burnt-orange gradient/utility layer (body background, `gradient-*`, `text-accent`, `.hero-tag`, `--accent-text`) that a token swap will **not** touch, and those classes are referenced across 42 component files. That, plus three undocumented helper tokens and a font-loader migration, is the real work the implementation story must own — the review below catches it before that story is written.

## 1. Mock ↔ spec fidelity — strong

### Findings

- **[low]** Every DESIGN.md `colors.light` / `colors.dark` triple, the shadow values, `accent-ink` (light `36 76% 33%` / dark `41 78% 63%`), and `footer-bg/fg` reproduce the mock `:root` blocks verbatim; `--radius 0.625rem`, Playfair+Inter, and the `clamp()` type scale also match. Fidelity of the captured tokens is not in question. *Fix:* none — this is the spec's strength.
- **[medium]** **Third helper token `--primary-ink` is used by the mocks but is entirely absent from DESIGN.md.** Both mocks style `.tag` text with `hsl(var(--primary-ink))` (dark `40 30% 92%`, light `223 47% 26%`), yet DESIGN.md's frontmatter and the "Semantic helpers" comment list only `accent-ink` and `footer-bg/fg`. An implementer reading only DESIGN.md would not create it. Impact is small (it's ~equal to `--primary` in both modes, so `hsl(var(--primary))` is a near-perfect substitute), but it's an uncaptured token. *Fix:* add `--primary-ink` to the helper list, or explicitly tell the implementer to use `--primary` for tag text and delete `--primary-ink` from the mock mental model.
- **[medium]** **The Tag description is only half-true across modes.** DESIGN.md §Components calls Tag a "Navy-tinted pill (`primary/.10`)". In dark mode `--primary` is inverted to off-white, so the mock tag is actually an *off-white*-tinted pill with off-white text — correct and readable, but not "navy". *Fix:* reword to "primary-tinted pill (navy in light, off-white in dark)".
- **[medium]** **Anchor table vs token table is internally contradictory.** DESIGN.md §Colors states Navy `#002251` = `215 100% 16%` and that it *is* `--primary` in light mode — but light `--primary` is `223 47% 26%` (different hue 223≠215, desaturated 47≠100, lighter 26≠16). No token anywhere uses the literal `#002251`/`215 100% 16%` or `#cb8a00`/`41 100% 40%`; the mode tokens are all tuned derivatives. This is fine as design intent but reads as a factual error that could make an implementer "correct" the tokens toward the raw anchors. *Fix:* label the anchor hexes as *reference/provenance only* and state the mode triples are the authoritative, tuned values.
- **[low]** **CTA-box ghost button override isn't a documented token.** The mock's secondary CTA button carries inline `border-color:hsl(var(--primary-foreground)/.3);color:hsl(var(--primary-foreground))` — a second ghost treatment distinct from the default `--foreground`/`--border` ghost in §Components. Needed so the "Send Email" button reads correctly on the inverted CTA panel in both modes. *Fix:* note the CTA secondary button as a ghost variant bound to `primary-foreground`.

## 2. Two-mode coherence — strong

### Findings

- **[low]** The inverted dark `--primary` (off-white) is handled consistently: `button-primary`, the full-bleed `.cta-box`, `.avatar`/`.portrait` gradients, and `.tag` all pair `--primary` with `--primary-foreground`, so the CTA panel cleanly flips navy (light) ⇄ off-white (dark) and the ghost-on-CTA stays legible in both. `button-accent` (gold bg / navy `accent-foreground` text) and `badge-eyebrow`/`icon-badge`/`stepnum` (gold + `accent-ink`) hold contrast in both modes. Verified against both screenshots — no combo reads wrong. *Fix:* none; call out in the story that the CTA panel is *intentionally* off-white in dark mode so no one "fixes" it as a bug.
- **[low]** DESIGN.md §Components documents only the default ghost button; the inverted-surface ghost (see 1) is the only mode combo not captured as a component spec. *Fix:* as above.

## 3. Values-swap portability — thin (this is where the port is NOT clean)

### Findings

- **[critical]** **The current `globals.css` is not a neutral shadcn baseline — it hardcodes the old burnt-orange (hue 25) look outside the token system, and 42 component files depend on it.** `body { background: var(--gradient-hero) }` paints an orange/blue gradient (mocks use `hsl(var(--background))`); `--gradient-hero/-text/-card/-accent`, `--accent-text: 25 85% 35%`, `.hero-tag` (`hsl(25 97% 71%)`), `.text-accent*` overrides, and `.text-accent-light` are all hue-25 orange and will **survive a token swap untouched** — directly violating DESIGN.md's "Don't reintroduce the old burnt-orange" rule. Grep shows `gradient-text` / `text-accent` / `hero-tag` / `shadow-glow` / `animate-float` / `font-body` used across 42 files (all `guides/*`, `Hero`, `Pricing`, `About`, `CTA`, `Services`, etc.). *Fix:* the port must (a) set `body` background to `hsl(var(--background))`, (b) purge or remap every `--gradient-*` and orange utility to the gold/navy tokens, and (c) audit all 42 consumers — this is a component-styling pass, not a numbers-only swap.
- **[high]** **The current `:root` is dark, not light.** Today `:root` and `.dark` are near-identical dark-navy palettes (app is dark-only). The spec requires `:root` = Warm Scholar **light** and `.dark` = Deep Authority. So the `:root` block must be *replaced wholesale* with the light palette — there is no existing light theme to preserve. Clean conceptually, but it's a full rewrite of both token blocks, not an edit of a few channels. *Fix:* replace both blocks verbatim from DESIGN.md frontmatter; add `--accent-ink`, `--primary-ink`, `--footer-bg`, `--footer-fg` to **both** `:root` and `.dark`.
- **[high]** **Three spec/mock tokens are non-native and must be added by hand:** `--accent-ink`, `--footer-bg`, `--footer-fg` (plus undocumented `--primary-ink`). They aren't in `globals.css` or wired in `tailwind.config.ts`. DESIGN.md flags accent-ink/footer as "add or inline" but the story must actually (a) declare them per-mode and (b) either add Tailwind color mappings or use `hsl(var(--accent-ink))` arbitrary values in components. Footer is dark-navy in **both** modes, so `--footer-bg/fg` do *not* invert — they carry near-identical dark values in `:root` and `.dark`. *Fix:* explicit token additions + decide Tailwind-mapped vs arbitrary.
- **[medium]** **next-themes is installed and wired, but its config contradicts "dark is default."** `providers.tsx` already has `<ThemeProvider attribute="class" defaultTheme="dark" enableSystem disableTransitionOnChange>`. Two conflicts with EXPERIENCE.md: (1) `enableSystem` means a first-time visitor on a light-OS gets **light** first paint, not the mandated dark default; (2) `disableTransitionOnChange` disables the very "colors cross-fade" the toggle interaction promises. *Fix:* set `enableSystem={false}` (or otherwise force dark as the true default) and drop `disableTransitionOnChange` if the cross-fade is wanted. Good news: the provider, `suppressHydrationWarning` on `<html>`, and `attribute="class"` are already in place, so no net-new theming plumbing — only config + the toggle UI.
- **[medium]** **The theme-toggle control and theme-driven logo swap are net-new.** No `ThemeToggle` / `setTheme` usage exists outside the provider; the Navbar has no sun/moon control. EXPERIENCE.md requires a keyboard-operable navbar toggle with `aria-label` reflecting the target mode, and DESIGN.md requires swapping reversed↔standard logo by surface/mode. *Fix:* build the toggle + a theme-aware logo component in the port story.
- **[low]** **Radius & button-radius port cleanly by coincidence — note it so no one "fixes" it.** `--radius` swaps `0.75rem`→`0.625rem`; `tailwind.config.ts` already derives `lg/md/sm` from `var(--radius)` exactly as the spec assumes. The mock's `calc(var(--radius)*.8)` button radius (~8px) equals shadcn's default `rounded-md` (`0.625rem − 2px` = 8px), so stock shadcn buttons match the mock with **no** component edit. *Fix:* none; document the equivalence.
- **[low]** **`sidebar-*` tokens are unspecified and off-brand; `chart-*`/`input`/`ring`/`secondary`/`popover` are fine.** The app carries generic-gray `--sidebar-*` (light `0 0% 98%` in `:root`) that DESIGN.md never addresses — harmless in Stage 1 (no sidebar renders) but off-brand if reused. `input`, `ring`, `secondary`, `popover` are all defined by the spec and present in the app; `chart-*` exist in neither and aren't needed. *Fix:* leave `sidebar-*` as-is for Stage 1; rebrand in Stage 2 if a portal sidebar appears.

## 4. Font migration — adequate (bigger than a string swap)

### Findings

- **[high]** **This is a font-loader migration, not just DM Sans→Inter.** The app currently loads fonts via a **Google Fonts CDN `@import`** at the top of `globals.css` (`DM Sans` + `Playfair Display`) and uses **no `next/font`** (nothing in `layout.tsx`). DESIGN.md mandates Playfair Display + **Inter** via **`next/font` self-hosted**. So the port must: (1) add `next/font/google` for Playfair Display + Inter in `layout.tsx`, exposing `--font-display` / `--font-body` CSS vars on `<body>`/`<html>`; (2) update `globals.css` `--font-body: 'DM Sans'`→`'Inter'`; (3) update `tailwind.config.ts` `fontFamily.body: ['DM Sans',…]`→`['Inter',…]`; (4) **delete** the CDN `@import` line. *Fix:* do all four; the string-swap alone (steps 2–3) would leave DM Sans still loading via the stale `@import`.

## Implementation-story checklist

Concrete work the later *port-into-acce-nextjs* story must include (derived from the gaps above):

- **Rewrite both token blocks** in `globals.css`: `:root` ← Warm Scholar **light** palette, `.dark` ← Deep Authority **dark** palette, copied verbatim from DESIGN.md frontmatter (space-separated HSL triples — format already matches).
- **Add the four non-native helper tokens** to *both* `:root` and `.dark`: `--accent-ink`, `--primary-ink`, `--footer-bg`, `--footer-fg` (footer values are dark-navy in both modes; they do not invert). Decide Tailwind color-mapping vs arbitrary `hsl(var(--…))` usage and apply consistently.
- **Purge the burnt-orange layer:** change `body` background to `hsl(var(--background))`; remove/remap `--gradient-hero/-text/-card/-accent`, `--accent-text`, `.hero-tag`, `.text-accent*`, `.text-accent-light`, and orange `--shadow-glow`; replace with the gold/navy tokens + the two mode shadows from DESIGN.md.
- **Audit all 42 component files** that use `gradient-text` / `text-accent` / `hero-tag` / `shadow-glow` / `animate-float` / `font-body` and re-point them to the new tokens (`accent-ink` for gold text/icons, `--accent` for fills only).
- **Swap `--radius`** `0.75rem`→`0.625rem`; confirm buttons stay on `rounded-md` (already ≈ mock's `*.8`); no button-component edit needed.
- **Font migration:** introduce `next/font/google` (Playfair Display + Inter) in `layout.tsx` with CSS-var output; update `--font-body` and Tailwind `fontFamily.body` to Inter; delete the CDN `@import`.
- **Fix next-themes config:** set `enableSystem={false}` so dark is the true first-paint default; remove `disableTransitionOnChange` if the color cross-fade is desired (per EXPERIENCE.md).
- **Build the theme toggle** (navbar sun/moon, keyboard-operable, `aria-label` = target mode) and a **theme-aware logo** component (reversed on dark/all footers, standard on light header).
- **Bring logo assets into the app:** copy `acce-logo-reversed.png` (reversed) and `acce-logo-transparent.png` (the "standard"/navy variant) from `design-artifacts/logo/` into `acce-nextjs/public/`; note the mock filenames (`logo-standard.png` / `logo-reversed.png`) differ from the on-disk names, and there is currently **no** brand logo in `public/` (only default Next SVGs + `priyanka.png`).
- **Replicate the CTA secondary-button ghost override** (`border/text = primary-foreground`) so it reads on the inverted CTA panel.
- **Verify contrast at build** for `muted-foreground` in both modes (dark `40 14% 70%`, light `223 12% 44%`) per the Accessibility Floor.
- **Leave `sidebar-*` tokens as-is** for Stage 1 (off-brand but unused); revisit in Stage 2.
