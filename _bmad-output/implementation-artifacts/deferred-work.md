## Deferred from: code review of story-2.2 (2026-07-05)

- **Display timezone relies on the deploy server's local TZ (cross-cutting with 2.1 input parsing)** — the admin classes list renders `start` via `toLocaleString("en-ZA")` with no explicit `timeZone` (`acce-nextjs/src/app/(admin)/admin/classes/page.tsx:60`). This is self-consistent today: Story 2.1's create form parses the `datetime-local` string with `z.coerce.date` (no offset → interpreted as server-local), so input and display share the same TZ frame and round-trip correctly. Forcing a display-only `timeZone: "Africa/Johannesburg"` here without also pinning 2.1's input parsing would introduce a mismatch (entered 10:00 → shown 12:00). Timezone handling is therefore a system-wide decision spanning 2.1 + this list (and echoes the 1.4 SAST/UTC comment deferral). Follow-up: decide the canonical app TZ once (likely `Africa/Johannesburg` end-to-end) and apply it consistently to both the create-form parse and every display edge. (Low/Medium; environmental, self-consistent today, not a code defect in isolation.)

## Deferred from: code review of story-3.3 (2026-07-06)

- **No `status`/`start` guard on the class-detail fetch (deep-link to non-SCHEDULED or past class)** — `(portal)/portal/classes/[id]/page.tsx` fetches the `GroupSession` by `id` only, with no `status: "SCHEDULED"` / `start > now` filter (unlike the 3.2 browse `findMany`). A student who deep-links / bookmarks `/portal/classes/<id>` for a `CANCELLED` (or already-started/past) class sees the full detail + a checkout panel — including, if balance is sufficient, the (inert) "Pay with balance" affordance. Only reachable by direct URL: the 3.2 browse links exclusively to SCHEDULED future classes. The button is inert in 3.3, and Story 3.4's `reserveSeat` will validate class state under the lock, so nothing can actually be booked. No AC in 3.3 requires 404-ing non-scheduled classes. Follow-up: when 3.4 lands the reserve flow, decide whether the detail page should also short-circuit non-SCHEDULED/past classes to a calm "no longer available" state (and/or `notFound()`), consistent with the reservation guard. (Low; deep-link-only, inert affordance, real enforcement lives in the 3.4 reserve path.)
- **Inert Pay CTA uses native `disabled` — not keyboard-focusable, contradicting the UX-DR6/NFR10 "keyboard-operable" note** — the forward-affordance button is rendered with the native `disabled` attribute (`page.tsx` "Pay with balance"), which removes it from the tab order; the inline comment "Button is keyboard-operable and perceivable even while disabled" is therefore inaccurate for a native disabled button (it is still announced/perceivable by screen readers as unavailable, but not focusable). The story's UX-DR6/NFR10 note asks the inert button to be "keyboard-operable with a visible focus ring." Transient: Story 3.4 converts this panel to a client island and makes the button live (no longer `disabled`), at which point it is naturally focusable. Follow-up: if the disabled affordance must be keyboard-focusable before 3.4, switch to `aria-disabled="true"` + an inert click handler (focusable-but-inert) instead of the native `disabled` attribute; otherwise close this when 3.4 wires the live button. (Low; transient, superseded by 3.4's client island.)
